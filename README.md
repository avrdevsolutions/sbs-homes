# starter-nextjs

> A no-feature starter template. Agents write all code from scratch. This repo encodes **architectural decisions** and **agent configuration** — not implementation.

---

## Quick Start

```bash
corepack enable && corepack prepare pnpm@latest --activate
pnpm install
pnpm dev --turbopack
# Open http://localhost:3000
```

**Requires**: Node.js 20+, pnpm 9+

---

## What Ships

Predefined scaffolding only — no example features:

- `src/middleware.ts` — security headers, request IDs
- `src/app/` — root layout, error boundaries, not-found page
- `src/lib/` — `cn()` helper, Zod-validated env config, `AppError` + `Result<T>` error handling
- `src/components/` — empty tier directories (ui / features / layout)
- `contracts/common.ts` — shared types: `ApiResponse<T>`, `Result<T>`, `ErrorCode`
- `tailwind.config.ts` — placeholder design tokens (must replace per project)
- 22 ADRs — the source of truth for all standards and decisions

Everything else is generated by agents when you describe what to build.

## Three Workflows

### `/design` — Visual exploration

Spawns 5 parallel `@designer` subagents (Gemini 2.5 Pro), each assigned a different aesthetic direction:

1. Brutalist
2. Luxury
3. Maximalist
4. Minimal
5. Retro-futuristic

Each designer outputs an HTML + CSS mockup with CSS custom properties defining the token palette. Human picks one. Then `@component-creator` translates the chosen mockup into React/Next.js components and updates `tailwind.config.ts` tokens.

**Use for**: landing pages, design system setup, marketing sections.
**Not for**: complex data features — use `/build-feature` instead.

### `/build-feature` — Full orchestration

```
Orchestrator (Sonnet 4.6) classifies task
  Planner (Sonnet 4.6) reads skills, writes SPEC.md with file ownership
    Designer (Gemini 2.5 Pro) creates HTML/CSS mockups
    Parallel execution:
        Component Creator + Page Assembler + API Builder (GPT-4.1)
    Test Writer (GPT-4.1) writes tests
    Parallel audit:
        Accessibility Auditor + Reviewer (Haiku 4.5)
```

The designer creates visual references. The planner runs **once** per feature. Execution agents translate spec + mockup to code mechanically.

### Individual agents — Standalone invocation

Any agent works without orchestration:

| Agent | Purpose |
|-------|---------|
| `@cto` | Architecture decisions, creates/updates ADRs |
| `@designer` | Visual mockups in HTML + CSS |
| `@reviewer` | ADR compliance check |
| `@component-creator` | Translate mockup/spec into React components |
| `@test-writer` | Write tests for existing code |
| `@accessibility-auditor` | WCAG audit |
| `@knowledge-sync` | Regenerate instructions + skills from ADRs |

## Architecture

### The Four Primitives

| Primitive | Trigger | Purpose | Location |
|-----------|---------|---------|----------|
| **Instructions** | Automatic (file-pattern scoped) | Design philosophy, ADR constraints | `.github/instructions/` |
| **Prompts** | Manual `/command` | Workflow invocation, model selection | `.github/prompts/` |
| **Skills** | On-demand by agents | Deep pattern knowledge for a domain | `.github/skills/` |
| **Agents** | Named invocation or subagent | Persistent persona, tool boundaries, model | `.github/agents/` |

### Knowledge Compilation Model

ADRs are verbose, human-written source of truth. Agents never read raw ADRs at task time. Instead:

```
CTO (Sonnet 4.6) writes/updates ADRs
  |
Knowledge Compiler (Sonnet 4.6, /sync-knowledge) reads ADRs, compiles output
  |
Instructions (auto-loaded, short)    Skills (on-demand, deeper)
  |                                    |
Planner reads skills, writes SPEC.md
  |
Execution agents read SPEC.md only
  |
Auditors check against instructions
```

**Rule**: Only put in context what agents cannot find themselves. ADR constraints are undiscoverable from code. Code patterns are discoverable.

### Model Assignment

| Agent | Model | Reason |
|-------|-------|--------|
| Designer | Gemini 2.5 Pro | Best visual/design output — creates HTML/CSS mockups |
| CTO | Claude Sonnet 4.6 | Architecture decisions, writes ADRs |
| Knowledge Compiler | Claude Sonnet 4.6 | Reads all ADRs, compiles instructions + skills |
| Orchestrator | Claude Sonnet 4.6 | Reasoning, delegation decisions |
| Planner | Claude Sonnet 4.6 | Deep research, unambiguous spec writing — runs ONCE per feature |
| Component Creator | GPT-4.1 | Translates mockups/specs to code — 0x cost, parallelizable |
| Page Assembler | GPT-4.1 | Same |
| API Builder | GPT-4.1 | Same |
| Test Writer | GPT-4.1 | Same |
| Accessibility Auditor | Claude Haiku 4.5 | Read-only, 0.33x cost |
| Reviewer | Claude Haiku 4.5 | Read-only, 0.33x cost |

**Cost pattern**: Sonnet thinks once, Gemini designs once, GPT-4.1 executes N times at **0x cost**. Haiku audits at 0.33x. Premium spend ≈ 3 requests per feature.

### The Spec-in-Markdown Pattern

The planner writes `.github/specs/<feature>.md` containing:

- Every file to create or modify
- Exact exports, function signatures, prop types
- Which agent owns which file (prevents merge conflicts)
- Edge cases and constraints from relevant skills

Execution agents read spec, translate to code mechanically. No reasoning required.

**Key**: File ownership in the spec makes parallelism safe. Two agents never touch the same file.

  .github/
    copilot-instructions.md       # ~20 lines, undiscoverable facts only
